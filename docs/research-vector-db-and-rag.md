# 技术调研：向量数据库对比 & RAG 架构演进

## 1. 向量数据库对比

| 维度 | Qdrant | Milvus | Weaviate | Chroma | pgvector | Vearch |
|------|--------|--------|----------|--------|----------|--------|
| 开源协议 | Apache 2.0 | Apache 2.0 | BSD-3 | Apache 2.0 | PG License | Apache 2.0 |
| 实现语言 | Rust | Go/C++ | Go | Python | C (PG扩展) | Go/C++ (Faiss) |
| 部署复杂度 | 低 | 高(依赖etcd/MinIO) | 中 | 极低(pip) | 极低(PG扩展) | 中(三组件) |
| 百万级性能 | 优秀 | 顶尖(十亿级) | 良好 | 一般(10万级) | 一般 | 良好 |
| 混合检索 | 稠密+稀疏(BM25/SPLADE) | 多索引+标量过滤 | 原生BM25+向量 | 不支持 | 需手动组合 | 向量+标量,无BM25 |
| LangChain集成 | 全覆盖 | 全覆盖 | 全覆盖 | 深度集成 | 支持 | 有限 |
| GitHub Stars | ~22k | ~33k | ~14k | ~18k | ~13k | ~2k |
| 适合场景 | 中大规模生产 | 超大规模企业级 | 知识图谱+语义搜索 | 快速原型/PoC | 已有PG基础设施 | 京东云生态 |

### Vearch 评估

**优点**：京东大促验证、基于 Faiss 性能扎实、分布式高可用
**缺点**：社区极弱(~2k stars)、无原生 BM25、LangChain 集成浅、迭代慢、英文生态几乎没有

**结论**：本项目继续使用 Qdrant，在中等规模 RAG 场景下性能、生态、易用性最优。

## 2. Demo RAG vs 生产级 RAG

| 维度 | 当前 Demo | 生产级 |
|------|----------|--------|
| 文档解析 | pymupdf4llm + markitdown | +OCR +表格提取 +图片描述 |
| 分块策略 | RecursiveCharacterTextSplitter 固定分块 | 语义分块 + 标题感知 + 元数据保留 |
| Embedding | 纯稠密向量(OpenAI) | 稠密+稀疏双路编码 |
| 检索 | 纯向量 cosine 相似度 | 混合检索(BM25+向量) → RRF融合 |
| Rerank | 无 | Cross-encoder reranker (+15~25%准确率) |
| Query改写 | 无 | HyDE / Multi-query / Step-back |
| 上下文压缩 | 无 | LLM 压缩去冗余 |
| 引用溯源 | 无 | chunk→原文页码/段落定位 |

### 生产级 RAG Pipeline

```
文档摄入(Offline):
  多格式文档 → 解析引擎(OCR+表格) → 智能分块 → 双路编码(稠密+稀疏) → 向量库+倒排索引

查询(Online):
  用户Query → Query改写(HyDE/Multi-Q) → 混合检索(BM25+向量→RRF融合)
  → Cross-Encoder Rerank → 上下文压缩 → LLM生成 → 引用溯源+高亮
```

### ROI 最高的改进路径

1. **加 Reranker**（最小改动,+15~25%准确率）
2. **混合检索**（BM25+向量融合,解决纯语义漏召回）
3. **Query 改写**（HyDE/Multi-query,改善query质量）
4. **引用溯源**（chunk映射回原文,提升用户信任）

## 3. RAG vs 长上下文 LLM (1M token)

### 核心问题
> LLM 上下文窗口已到 1M，直接把资料扔进去不就行了？

### 多维度对比

| 维度 | 长上下文 LLM (1M) | RAG |
|------|-------------------|-----|
| **成本** | $2-10/次 | ~$0.0001/次 (约1/1250) |
| **准确性** | Lost in the Middle: 中间信息丢失30%+ | 精准注入top-K chunk,信息密度高 |
| **延迟** | 首token数十秒 | 1-2秒 |
| **数据规模** | 上限3-5本书(~100篇论文) | 理论无上限,十亿级向量成熟方案 |
| **引用溯源** | 黑盒,无法确定来源 | chunk级定位,可跳转原文 |
| **增量更新** | 每次全量拼装 | 增量索引,秒级入库 |

### 关于"好用户能自己摘信息"

工具的价值恰恰是降低门槛。即使是专家,面对50篇论文全文也不可能秒级定位交叉验证段落。
更重要的是：长上下文中 LLM 自己都会 Lost in the Middle,何况人类用户。

### 结论：互补而非替代

```
海量文档库 (10M+ tokens)
  → RAG 检索层(粗筛最相关 Top-K)
    → 长上下文 LLM(精选上下文中深度推理)
      → 高质量答案 + 引用来源
```

**最佳实践：RAG 做大海捞针,长上下文做精读理解。**
